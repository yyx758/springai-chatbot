# 双令牌（JWT + Refresh Token）设计详解

## 一、为什么需要双 Token？

### 单 Token 的困境

假设只用一个 JWT Token，面临两难：

| Token 有效期 | 优点 | 缺点 |
|-------------|------|------|
| **短**（如 30 分钟） | 安全，泄露窗口小 | 用户每 30 分钟要重新登录，体验差 |
| **长**（如 7 天） | 用户体验好 | 一旦泄露，攻击者有 7 天的攻击窗口，且无法吊销 |

**核心矛盾：安全性 vs 用户体验，不可兼得。**

### 双 Token 的解决方案

```
┌──────────────────────────────────────────────────────────┐
│                     双 Token 分工                          │
│                                                            │
│   Access Token（通行证）          Refresh Token（续命卡）    │
│   ├── 有效期短：30 分钟           ├── 有效期长：7 天          │
│   ├── 每次请求携带                ├── 仅在刷新时使用          │
│   ├── 无状态验证（JWT 自包含）     ├── 有状态验证（查 Redis）   │
│   └── 泄露影响小（很快过期）       └── 服务端可控（可主动吊销） │
└──────────────────────────────────────────────────────────┘
```

**核心思想：用短令牌保安全，用长令牌保体验。**

- 日常请求使用 Access Token → 无状态、高效
- Access Token 过期后用 Refresh Token 换新的 → 用户无感知
- Refresh Token 存在 Redis → 服务端随时可以吊销

## 二、为什么 Access Token 用 JWT？

### 2.1 什么是 JWT

JWT（JSON Web Token）是一种**自包含**的令牌格式：

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsInVzZXJuYW1lIjoiYWRtaW4ifQ.xxxxx
│       Header（算法）       │         Payload（载荷）              │  签名  │
```

**解码 Payload 即可获得用户信息：**
```json
{
  "sub": "1",
  "username": "admin",
  "role": "ADMIN",
  "iat": 1716182400,
  "exp": 1716184200
}
```

### 2.2 为什么 Access Token 选 JWT

#### 原因 1：无状态验证，性能高

```
每个 API 请求都要验证 Token，对比两种方式：

【有状态方案】每次请求查数据库/Redis
客户端请求 → 查 Redis 验证 → 返回结果
              ↑
              每次都查，高并发下 Redis 压力大

【JWT 方案】本地验签即可
客户端请求 → 用密钥验签 + 解析 claims → 返回结果
              ↑
              不需要查任何存储，纯 CPU 计算
```

对于聊天接口这种高频调用场景，**无状态验证**优势明显。

#### 原因 2：自包含用户信息

```java
// 从 JWT 中直接解析用户信息，无需查数据库
Long userId = jwtTokenProvider.parseUserId(token);   // 提取 userId
String role = jwtTokenProvider.parseRole(token);       // 提取 role
```

**好处：**
- 减少数据库查询，降低延迟
- 权限校验（RBAC）可以直接从 Token 中获取角色

#### 原因 3：天然防篡改

```
Token = Header.Payload.Signature

签名 = HMAC-SHA256(Header.Payload, 密钥)

攻击者修改 Payload → 签名不匹配 → 验证失败
```

后端只需保管好密钥，就能确保 Token 不被篡改。

#### 原因 4：跨服务共享

如果未来拆分微服务，JWT 可以在多个服务之间共享验证，无需每个服务都连 Redis。

### 2.3 JWT 的缺点（为什么还需要 Refresh Token）

| 缺点 | 说明 |
|------|------|
| **无法主动失效** | JWT 签发后在有效期内一直有效，服务端无法提前吊销 |
| **体积较大** | 包含 Header + Payload + Signature，比随机字符串大 |
| **Payload 不加密** | Base64 解码即可看到内容，敏感信息不能放进去 |

这些缺点恰好被 Refresh Token 补齐。

## 三、为什么 Refresh Token 用随机 UUID？

### 3.1 两种方案对比

```
【JWT 作为 Refresh Token】
Token = eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsImV4cCI6MTcyMDAwMDAwMH0.xxxxx
         ↑ 自包含，有效期写死在里面

问题：
- 无法提前失效（除非换签名密钥，影响所有用户）
- 有效期在 Token 内部，服务端无法灵活控制

【随机 UUID 作为 Refresh Token】
Token = a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6...
         ↑ 无意义随机字符串，只是一个 Key

优势：
- 存储在 Redis，服务端完全控制
- 删除即失效，灵活吊销
```

### 3.2 为什么选随机 UUID

#### 原因 1：服务端完全可控

```java
// 存储：Token → userId 的映射，7 天过期
redisTemplate.opsForValue().set("refresh_token:" + token, userId, 7, TimeUnit.DAYS);

// 验证：查 Redis，存在则有效
Object userId = redisTemplate.opsForValue().get("refresh_token:" + token);

// 吊销：直接删除，立即失效
redisTemplate.delete("refresh_token:" + token);

// 全部吊销：删除该用户的所有 Refresh Token
revokeAllForUser(userId);
```

**JWT 做不到这些**，因为 JWT 签发后就"失控"了。

#### 原因 2：一次性消费（Token 轮转）

```java
public Long getUserIdAndInvalidate(String token) {
    // getAndDelete：原子操作，读完就删
    Object userId = redisTemplate.opsForValue().getAndDelete("refresh_token:" + token);
    return userId != null ? Long.valueOf(String.valueOf(userId)) : null;
}
```

```
正常流程：
  用户带着 RT1 来刷新 → 读取 userId → 删除 RT1 → 发放 AT2 + RT2

攻击场景：
  攻击者拿着泄露的 RT1 再次刷新 → RT1 已被删除 → 返回 null → 刷新失败
```

**如果是 JWT**，无法实现"用一次就作废"，因为 JWT 的有效期是固定的。

#### 原因 3：无信息泄露风险

```
JWT：Base64 解码就能看到 userId、role 等信息
UUID：a1b2c3d4... 无任何语义信息，泄露也看不出什么
```

#### 原因 4：体积更小

```
JWT：约 100-200 字节（Header.Payload.Signature）
UUID × 2：64 字节（两个 UUID 拼接）
```

### 3.3 为什么是 UUID × 2？

```java
public String createRefreshToken() {
    return UUID.randomUUID().toString().replace("-", "")
         + UUID.randomUUID().toString().replace("-", "");
    // 32 + 32 = 64 位十六进制字符
    // 碰撞概率：16^64 分之一，几乎不可能
}
```

**用两个 UUID 拼接的原因：** 增加随机性，降低碰撞概率。即使一个 UUID 有缺陷，两个拼接后安全性更强。

## 四、完整工作流程

### 4.1 登录流程

```
客户端                          服务端                        Redis
  │                               │                            │
  │  POST /api/auth/login         │                            │
  │  { username, password }       │                            │
  │ ─────────────────────────────>│                            │
  │                               │  验证用户名密码              │
  │                               │  生成 Access Token (JWT)    │
  │                               │  生成 Refresh Token (UUID)  │
  │                               │ ──────────────────────────>│
  │                               │  SET refresh_token:abc → 1  │
  │                               │  EXPIRE 7天                 │
  │                               │<────────────────────────── │
  │  { token, refreshToken }      │                            │
  │ <─────────────────────────────│                            │
  │                               │                            │
  │  localStorage 存储两个 Token   │                            │
```

### 4.2 正常请求流程

```
客户端                          服务端
  │                               │
  │  GET /api/chat/records        │
  │  Header: Bearer {JWT}         │
  │ ─────────────────────────────>│
  │                               │  用密钥验签 JWT
  │                               │  解析 userId, role
  │                               │  权限校验（@RequireRole）
  │  { data }                     │
  │ <─────────────────────────────│
```

**不查 Redis，纯本地验签，高效。**

### 4.3 Token 刷新流程

```
客户端                          服务端                        Redis
  │                               │                            │
  │  请求返回 401                  │                            │
  │  POST /api/auth/refresh       │                            │
  │  { refreshToken }             │                            │
  │ ─────────────────────────────>│                            │
  │                               │  GETDEL refresh_token:abc  │
  │                               │ ──────────────────────────>│
  │                               │  返回 userId = 1（并删除）   │
  │                               │<────────────────────────── │
  │                               │                            │
  │                               │  生成新 AT + 新 RT           │
  │                               │ ──────────────────────────>│
  │                               │  SET refresh_token:def → 1  │
  │                               │<────────────────────────── │
  │  { token, refreshToken }      │                            │
  │ <─────────────────────────────│                            │
  │                               │                            │
  │  更新 localStorage             │                            │
```

### 4.4 踢人流程（密码重置）

```
用户重置密码                      服务端                        Redis
  │                               │                            │
  │  POST /api/auth/reset-password│                            │
  │ ─────────────────────────────>│                            │
  │                               │  修改密码哈希                │
  │                               │  查找该用户所有 RT            │
  │                               │  DEL refresh_token:abc      │
  │                               │  DEL refresh_token:def      │
  │                               │  DEL refresh_token:ghi      │
  │                               │ ──────────────────────────>│
  │                               │  全部删除                    │
  │                               │<────────────────────────── │
  │  { success }                  │                            │
  │ <─────────────────────────────│                            │
  │                               │                            │
  │  所有设备的 RT 失效 →           │                            │
  │  所有设备必须重新登录            │                            │
```

## 五、安全设计总结

| 安全特性 | 实现方式 | 对应代码 |
|---------|---------|---------|
| **Token 防篡改** | JWT 签名（HMAC-SHA256） | `JwtTokenProvider.createAccessToken()` |
| **短有效期** | Access Token 30 分钟 | `token-expire-ms: 1800000` |
| **长有效期** | Refresh Token 7 天 | `refresh-token-expire-ms: 604800000` |
| **Token 轮转** | 刷新时 getAndDelete 旧 RT | `RefreshTokenStore.getUserIdAndInvalidate()` |
| **防重放** | 旧 RT 一次性消费 | `redisTemplate.opsForValue().getAndDelete()` |
| **强制踢人** | 删除用户所有 RT | `RefreshTokenStore.revokeAllForUser()` |
| **无状态高效验证** | JWT 本地验签 | `JwtTokenProvider.parseUserId()` |
| **服务端可控** | RT 存 Redis，随时吊销 | `RefreshTokenStore.store()` |

---

## 六、为什么不选其他方案？

### 方案对比

| 方案 | 说明 | 问题 |
|------|------|------|
| **Session + Cookie** | 传统 Web 方案 | 不适合前后端分离、跨域困难、集群需共享 Session |
| **单个长 JWT** | 只用一个 JWT | 无法主动吊销、泄露风险大 |
| **单个短 JWT** | 只用一个短 JWT | 用户频繁登录，体验极差 |
| **JWT + Redis 黑名单** | JWT 存在，Redis 存吊销列表 | 每次还是查 Redis，失去 JWT 无状态的意义 |
| **JWT + UUID Refresh** | 本项目的方案 | ✅ 各取所长 |

### 最终选择

```
Access Token = JWT    → 无状态、高频验证、自包含用户信息
Refresh Token = UUID  → 有状态、服务端可控、支持吊销和轮转
```

**一句话总结：用 JWT 保效率，用 Redis 保安全，两者互补。**
