# AI Studio 智能客服项目面试复习文档

## 1. 项目一句话介绍

这是一个基于 Spring Boot 3、Spring AI、MyBatis-Plus 的智能客服系统，支持 DeepSeek 和 Ollama 多模型对话、SSE 流式输出、图片加文本的多模态问答、RAG 知识库增强、JWT 双令牌认证、RBAC 权限管理、Kafka 异步事件持久化、Redis 缓存、Spring Cloud Gateway 网关、Nacos 服务注册发现和 Docker Compose 一键部署。

面试开场可以这样说：

> 我做的是一个 AI Studio 智能客服系统，不只是简单调大模型 API，而是把 AI 对话、知识库增强、用户认证、文件管理和微服务工程化都串起来了。系统拆成 chatbot-service、file-service 和 gateway 三个模块，通过 Kafka 做异步事件通信，通过 Redis 做会话缓存和 Refresh Token 存储，通过 Nacos 做服务注册发现，最后用 Docker Compose 编排 MySQL、Redis、Kafka、Nacos 和各个应用服务。

## 3. 面试最值得讲的三条主线

## 主线一：AI 对话不是简单调接口，而是多模型、多模态、RAG 的组合

### 3.1 可以怎么讲

这个项目的 AI 能力主要有三层：

1. 多模型支持：通过 Spring AI 同时接入 DeepSeek 云端模型和 Ollama 本地模型，用户可以在运行时选择模型。
2. 多模态支持：当用户上传图片并提问时，系统会自动切换到 Ollama 视觉模型，例如 llava，通过 Spring AI 的 Media 消息把图片和文本一起传给模型。
3. RAG 检索增强：用户可以上传知识库文档，提问时系统会检索当前用户自己的知识文档，召回 Top-K 片段，把知识片段作为额外的 SystemMessage 注入到 prompt 里，让回答更贴合私有知识。

面试表达：

> 我没有只做一个大模型接口封装，而是把大模型能力做成了可切换、可降级的对话链路。普通文本对话可以走 DeepSeek 或 Ollama；如果请求带图片，就强制切换到 Ollama 视觉模型；如果用户开启 RAG，会先从自己的知识库里召回相关片段，再把知识片段注入 prompt。整个回答通过 SSE 流式返回，用户前端能实时看到模型输出。

### 3.3 你要能解释的设计点

#### 为什么用 SSE？

因为大模型回答可能比较慢，如果等完整结果生成完再返回，用户体验很差。SSE 可以边生成边返回，前端能做打字机效果，而且对服务端来说比 WebSocket 更轻量，适合单向流式输出。

#### 为什么 Ollama 要加 Semaphore？

本地模型推理资源有限，尤其是小机器或远程服务器部署时，并发请求太多会导致模型响应超时甚至崩溃。所以代码里用 `Semaphore(1)` 控制 Ollama 单线程排队，当前一个请求正在处理时，后续请求会进入 queued 状态，最多等待 120 秒。

可以这样说：

> DeepSeek 这种云模型可以依赖服务商承载并发，但 Ollama 是我自己的本地推理服务，资源有限，所以我对 Ollama 请求做了排队控制，避免并发打爆模型服务。

#### RAG 是向量检索吗？

不是。当前版本是轻量级关键词评分召回。

你要诚实回答：

> 当前版本没有引入向量数据库，而是用关键词评分做轻量 RAG。原因是项目初期知识库规模不大，关键词召回实现简单、成本低、可解释性强。后续如果知识库规模变大，我会升级到 embedding + vector store，比如 pgvector、Milvus 或 Elasticsearch dense vector。

#### RAG 检索失败怎么办？

`ChatbotService` 里对 RAG 检索包了 try-catch，失败会降级成普通对话，不影响用户提问主流程。

可以这样说：

> RAG 是增强能力，不应该成为聊天链路的单点故障，所以我做了降级处理。检索失败时记录日志，然后按普通对话继续。

## 主线二：Kafka 异步事件持久化和可靠性设计

### 4.1 可以怎么讲

聊天服务生成回答后，没有直接同步写数据库，而是发送 `ChatEvent` 到 Kafka，由消费者异步写入 MySQL，并删除 Redis 聊天历史缓存。

这样做的收益：

1. 降低用户请求链路耗时：模型回答结束后不用阻塞等待数据库写入。
2. 解耦聊天生成和聊天记录持久化。
3. 支持多实例扩展：后续 chatbot-service 横向扩展时，Kafka 消费组可以分担持久化任务。
4. 可靠性更强：消费者手动 ACK，失败重试，重试耗尽后进死信队列。

面试表达：

> 一开始聊天记录可以直接同步写 MySQL，但这样会把数据库写入耦合到聊天响应链路里。后来我把它改成事件驱动：模型完整回答后发送 ChatEvent 到 Kafka，消费者负责持久化。消费者关闭自动提交，只有 MySQL 写入成功、Redis 缓存失效完成后才手动 ACK。失败时交给 DefaultErrorHandler 自动重试，重试耗尽进入 DLT 死信队列。

### 4.3 你要能解释的设计点

#### 为什么不用直接 `@Async` 写数据库？

可以回答：

> `@Async` 只能解决当前 JVM 内异步执行，服务重启或线程池异常时任务可能丢失，也不方便多实例扩展。Kafka 是持久化消息队列，可以把事件保存下来，消费者失败还可以重试和进入死信队列，更适合服务间异步通信。

#### 为什么消费者要手动 ACK？

因为只有业务真正处理成功之后才能确认消息。自动提交 offset 可能会出现一种情况：offset 已提交，但 MySQL 写入失败，消息就丢了。

可以回答：

> 我关闭了自动提交，使用 `MANUAL_IMMEDIATE` 手动 ACK。消费者先写 MySQL，再删除 Redis 缓存，全部成功后才 `ack.acknowledge()`。如果写库失败，异常会抛给 `DefaultErrorHandler`，Kafka 会重试而不是直接提交 offset。

#### 为什么写 MySQL 后删除 Redis，而不是更新 Redis？

这是 Cache-Aside 缓存失效策略。

可以回答：

> 聊天历史是以 MySQL 为准，Redis 是缓存。消费者写库成功后删除对应 session 的 Redis 缓存。下一次读取历史时，如果 Redis miss，就从 MySQL 读最新数据并回填 Redis。相比直接更新缓存，删除缓存更简单，也能减少写缓存失败导致的数据不一致问题。

#### DLT 是什么？

DLT 是 Dead Letter Topic，死信队列。

可以回答：

> 消费者处理失败后会自动重试 3 次，每次间隔 1 秒。如果还是失败，就进入 `chat.events.DLT`。生产环境可以进一步把 DLT 消息写入审计表，或者发告警通知运维处理。

#### 项目现在能保证消息绝对不丢吗？

成熟回答：

> 不能说绝对不丢。当前已经做了生产者重试、消费者手动 ACK、消费失败重试和 DLT。但如果生产者 3 次发送都失败，目前只是记录日志。生产级可以继续引入本地消息表或 Outbox Pattern，先把待发送事件落库，再由后台任务补偿发送到 Kafka。

这个回答很重要，说明你知道自己项目的边界。

### 4.4 Kafka 单测可以怎么讲

`KafkaReliabilityTest` 覆盖了：

1. 生产者第一次发送成功。
2. 生产者前两次失败，第三次成功。
3. 生产者三次都失败但不抛异常。
4. 消费者正常写 MySQL、删除 Redis、手动 ACK。
5. MySQL 写入失败时不 ACK，让错误处理器接管。
6. Redis 删除失败不影响主流程。
7. 清洗 SSE 协议残留。
8. 图片数据兼容 fileKey 和 Base64 两种模式。

面试表达：

> Kafka 这块我专门写了 Mockito 单测，不依赖真实 Kafka，把 producer retry、consumer ACK、Redis 缓存删除、异常传播和图片数据兼容都覆盖了。

## 主线三：认证、安全、微服务拆分和部署闭环

### 5.1 可以怎么讲

项目不是单体 demo，而是做了微服务拆分和基础安全治理：

1. Gateway 作为统一入口，负责 JWT 鉴权和管理员接口权限控制。
2. chatbot-service 内部也有拦截器和 `@RequireRole` 注解，做接口级 RBAC。
3. JWT 使用 Access Token + Refresh Token 双令牌机制。
4. Refresh Token 存在 Redis，并且刷新时使用 `getAndDelete` 做一次性轮换。
5. 重置密码、禁用用户、删除用户时，会撤销该用户全部 Refresh Token，强制重新登录。
6. file-service 独立管理文件和知识库文档，聊天记录只保存 fileKey，避免把大文件塞到聊天主服务里。
7. Docker Compose 一键启动 MySQL、Redis、Kafka、Nacos、chatbot-service、file-service、gateway。

面试表达：

> 我把系统拆成了主聊天服务、文件服务和网关服务。网关统一做 JWT 校验，把 userId、username、role 注入 Header 传给下游。业务服务内部也有认证拦截器和角色注解，管理员接口需要 ADMIN 角色。Token 设计上使用短期 Access Token 加长期 Refresh Token，Refresh Token 存 Redis，并且刷新时会立刻失效旧 token，避免被重复使用。

| `chatbot-service/src/main/java/com/example/chatbot/security/AuthInterceptor.java` | 主服务接口认证、`@RequireRole` 权限检查 |

### 5.3 你要能解释的设计点

#### 为什么要双令牌？

Access Token 有效期短，泄露风险低；Refresh Token 有效期长，用于无感刷新。

可以回答：

> Access Token 用于日常接口鉴权，有效期较短。Refresh Token 用来换新的 Access Token，放在 Redis 里便于服务端主动撤销。这样既能保证用户体验，也能在密码重置、账号禁用时强制所有设备重新登录。

#### 为什么 Refresh Token 刷新时要 `getAndDelete`？

为了做 Refresh Token 轮换，防止同一个 Refresh Token 被重复使用。

可以回答：

> 每次刷新时，我会从 Redis 里 `getAndDelete` 旧 Refresh Token，拿到 userId 后重新签发新的 Access Token 和 Refresh Token。旧 token 立刻失效，如果被截获也不能重复刷新。

#### 为什么网关和服务内部都做鉴权？

可以回答：

> Gateway 是统一入口，负责第一层鉴权和路由前校验；服务内部拦截器是第二层保护，避免绕过网关直连服务时没有权限控制。实际生产中也可以通过网络隔离限制服务只接受网关流量。

#### file-service 为什么独立出来？

可以回答：

> 图片和文档文件属于独立资源，和聊天主流程耦合度不高，但会占用存储和处理资源。拆成 file-service 后，聊天服务只关心 fileKey，不直接存大图片 Base64，后续也方便把本地存储切换成 MinIO 或对象存储。

#### Docker Compose 的意义是什么？

可以回答：

> 项目依赖 MySQL、Redis、Kafka、Nacos 和三个应用服务，如果手动配置很繁琐。我用 Docker Compose 把基础设施和应用服务编排起来，并配置了 healthcheck 和 depends_on，方便本地演示和部署。

## 6. 面试高频追问和推荐回答

### 6.1 你这个项目最大的难点是什么？

推荐回答：

> 最大难点不是单个接口，而是把 AI 能力和工程链路结合起来。比如聊天要支持流式输出、多模型切换、多模态输入、RAG 增强，同时还要考虑历史上下文、缓存、异步持久化和失败降级。尤其是 Kafka 改造后，我需要处理消息可靠性、消费者 ACK、缓存一致性和死信队列，这些比单纯调模型 API 更接近真实后端项目。

### 6.2 为什么要用 Redis？

推荐回答：

> Redis 主要用在三个地方。第一是缓存聊天历史，构建上下文时优先从 Redis 读，miss 时再读 MySQL 并回填；第二是存 Refresh Token，方便服务端主动撤销；第三是验证码缓存，控制验证码有效期和验证逻辑。

### 6.3 Redis 和 MySQL 的一致性怎么保证？

推荐回答：

> 以 MySQL 为准，Redis 只是缓存。聊天记录由 Kafka 消费者写入 MySQL，写入成功后删除对应 session 的 Redis 缓存。下一次用户读取历史时从 MySQL 重新加载并回填 Redis。这是一种 Cache-Aside 策略。

### 6.4 如果 Kafka 消费失败怎么办？

推荐回答：

> 消费者不手动 catch 核心业务异常，而是让异常抛给 `DefaultErrorHandler`。它会按配置重试 3 次，每次间隔 1 秒。重试耗尽后通过 `DeadLetterPublishingRecoverer` 发送到 DLT。消费者只有成功写库并处理缓存后才 ACK。

### 6.5 如果 AI 服务不可用怎么办？

推荐回答：

> 构造 ChatModel Bean 时使用 `ObjectProvider`，并对模型不可用做 try-catch，避免某个模型服务不可用导致整个应用启动失败。运行时如果用户指定的模型不可用，会尝试 fallback 到另一个模型；如果都不可用，则通过 SSE 返回错误事件。

### 6.6 DeepSeek 和 Ollama 怎么选择？

推荐回答：

> DeepSeek 是云端模型，效果和稳定性更好，适合普通文本对话；Ollama 是本地模型，适合本地部署和隐私场景。图片多模态目前走 Ollama 视觉模型，比如 llava。因为 Ollama 推理资源有限，所以我加了信号量限制并发。

### 6.7 为什么没有直接用 Spring Security？

推荐回答：

> 这个项目认证逻辑比较明确，主要是 JWT 和简单 RBAC，所以我自己实现了拦截器和注解式权限控制，便于理解和展示底层流程。生产项目如果权限体系更复杂，可以换成 Spring Security，把认证、授权、过滤器链交给框架管理。

### 6.8 项目有什么不足？后续怎么优化？

推荐回答：

> 主要有几个优化方向。第一，RAG 现在是关键词评分，后续可以升级为向量检索。第二，Kafka producer 最终失败目前只是记录日志，生产级可以引入本地消息表或 Outbox Pattern。第三，Refresh Token 撤销所有用户 token 时用了 Redis keys 扫描，数据量大时可以改成按 userId 维护 token set 或 token version。第四，配置里的本地密码和默认 JWT secret 需要在生产环境通过环境变量或密钥管理系统替换。

## 7. 30 秒版本项目介绍

> 我做了一个 AI Studio 智能客服系统，基于 Spring Boot 3 和 Spring AI，支持 DeepSeek 和 Ollama 多模型聊天、SSE 流式输出、图片加文本多模态问答，以及基于用户知识库的 RAG 增强。工程上我把系统拆成 chatbot-service、file-service 和 gateway，通过 Redis 做聊天历史缓存和 Refresh Token 存储，通过 Kafka 做聊天记录异步持久化，并配置了手动 ACK、失败重试和死信队列。最后用 Nacos 做服务注册发现，用 Docker Compose 编排 MySQL、Redis、Kafka、Nacos 和应用服务。

## 8. 2 分钟版本项目介绍

> 这个项目是一个智能客服系统，核心目标是把大模型能力落到一个完整的后端工程里。AI 层面，我用 Spring AI 同时接入 DeepSeek 和 Ollama，普通文本可以选择模型，图片加文本会自动切换到 Ollama 的视觉模型。聊天接口使用 SSE 流式返回，用户能实时看到模型输出。为了增强回答的准确性，我做了一个轻量 RAG，用户可以上传自己的知识文档，提问时系统会根据关键词评分召回 Top-K 片段，并把片段注入 prompt。
>
> 工程层面，系统拆成三个服务：chatbot-service 负责聊天、认证和知识库；file-service 负责文件上传、缩略图和知识库文档解析；gateway 负责统一入口和 JWT 鉴权。聊天历史优先从 Redis 读取，miss 后从 MySQL 读取并回填。聊天完成后不是同步写库，而是发送 ChatEvent 到 Kafka，由消费者异步持久化到 MySQL，并删除 Redis 缓存。Kafka 消费者使用手动 ACK，失败交给 DefaultErrorHandler 重试，重试耗尽进入 DLT。
>
> 安全上，我实现了 Access Token + Refresh Token 双令牌，Refresh Token 存 Redis，刷新时用 getAndDelete 做一次性轮换，密码重置或账号禁用时会撤销该用户所有 Refresh Token。部署上用 Docker Compose 编排 MySQL、Redis、Kafka、Nacos、Gateway 和两个业务服务。

## 9. 按代码复习顺序

建议你按这个顺序看代码，不要从 controller 随机看：

1. `docker-compose.yml`
   先看整个项目依赖哪些服务，理解 MySQL、Redis、Kafka、Nacos、chatbot-service、file-service、gateway 怎么启动。

2. `pom.xml` 和各模块 `pom.xml`
   看项目是 Maven 多模块，确认 Spring AI、Kafka、Nacos、JWT、Redis、Flyway 这些依赖从哪里来。

3. `ChatbotService.java`
   重点看模型选择、RAG 注入、Redis/MySQL 历史、SSE 流式输出、多模态。

4. `RagService.java`
   看知识库文档如何按关键词评分召回，以及如何构造知识 prompt。

5. `ChatEventProducer.java`、`ChatEventConsumer.java`、`KafkaConsumerConfig.java`
   看 Kafka 从发送到消费到 ACK 到 DLT 的完整链路。

6. `KafkaReliabilityTest.java`
   看你怎么证明 Kafka 可靠性逻辑被测试过。

7. `AuthService.java`、`RefreshTokenStore.java`、`AuthInterceptor.java`
   看登录、刷新 token、撤销 token、角色校验。

8. `AuthGlobalFilter.java`
   看网关如何解析 JWT、校验管理员路径、注入用户 Header。

9. `FileService.java`
   看文件上传、图片压缩缩略图、知识库文档解析、fileKey 设计。

> AI Studio 智能客服系统 | Spring Boot 3, Spring AI, Kafka, Redis, MySQL, Nacos, Docker

## 11. 最后提醒

面试时不要把这个项目讲成“我用了很多技术栈”。重点要讲：

1. 为什么这么设计。
2. 每个技术解决了什么问题。
3. 当前方案有什么边界。
4. 如果上生产，下一步怎么优化。

尤其要注意：

| 问题 | 正确说法 |
| --- | --- |
| RAG 是不是向量检索 | 当前是关键词评分召回，后续可升级向量检索 |
| Kafka 是否绝对不丢 | 当前有重试、手动 ACK、DLT，但 producer 最终失败还需 Outbox 补偿 |
| 配置里有本地密码 | 本地开发配置，生产通过环境变量和密钥管理替换 |
| 为什么不用 Spring Security | 当前权限简单，自定义实现便于理解；复杂生产场景可迁移 Spring Security |

你最应该背熟的三句话：

1. 我不是简单调大模型 API，而是做了多模型、多模态、RAG 和 SSE 流式输出的完整聊天链路。
2. 我用 Kafka 把聊天生成和聊天记录持久化解耦，并做了生产者重试、消费者手动 ACK、失败重试和死信队列。
3. 我把系统拆成主服务、文件服务和网关服务，并用 JWT 双令牌、Redis、Nacos 和 Docker Compose 做了完整工程闭环。
