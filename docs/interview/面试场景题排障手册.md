# AI Studio 场景题与排障手册

这份文档用于面试复盘和真实排障。每个场景都按“现象、怎么发现、怎么定位、怎么解决、面试回答”组织，方便把项目经历讲成可验证的工程能力。

## 1. 如果发现 RAG 召回率不佳怎么办？

### 现象

用户问了知识库里应该有的内容，但回答没有引用相关文档，或者召回结果不相关。

常见表现：

- 明明上传过 Redis 文档，问 `Redis Lua` 没命中。
- 问 `Gateway Nacos`，结果召回了无关文档。
- LLM 回答像通用知识，没有基于知识库。

### 怎么发现

不要直接看最终回答，要先看检索结果。

当前项目可以分层看：

- ES 关键词召回：查 `ai_studio_knowledge_v3`。
- MySQL FULLTEXT fallback：查 `MATCH(title, content, tags) AGAINST (...)`。
- PGVector 向量召回：看 `VectorRagService` 日志。
- RRF 融合：看最终 TopK 是否包含目标 chunk。

相关代码：

- `chatbot-service/src/main/java/com/example/chatbot/service/RagService.java`
- `chatbot-service/src/main/java/com/example/chatbot/rag/HybridSearchService.java`
- `chatbot-service/src/main/java/com/example/chatbot/rag/ElasticsearchKeywordSearchService.java`
- `chatbot-service/src/main/java/com/example/chatbot/rag/RrfFusionService.java`

### 怎么定位

按顺序排：

1. 文档是否入库：`knowledge_document` 有没有数据，`enabled=1`。
2. ES 是否有索引：`GET /ai_studio_knowledge_v3/_count`。
3. ES 是否按用户隔离：查询里必须有 `userId` filter。
4. query 是否被过度改写：看 `QueryEnhancer` 增强后的 query。
5. 是否 chunk 切得太粗或太碎。
6. 是否字段权重不合理，例如 content 权重过高导致正文噪声召回。
7. 是否 `minimum_should_match` 太松或太严。

### 怎么解决

可分两层。

关键词层：

- 调整 ES 字段权重：`title^5`、`keywords^4`、`tags^3`、`summary^2`、`content^1`。
- 减少正文单字误召回：standard multi_match 不直接扫全文 content。
- 给 ngram 子字段加更高 `minimum_should_match`。
- 对标题、标签、关键词提高 boost。
- 增加领域同义词：`query-synonyms.yml`。
- 对不存在主题允许返回空，不要为了召回率硬塞无关内容。

向量层：

- 检查 PGVector 是否启用。
- 检查 embedding 维度是否和表结构一致。
- 检查 query 和 chunk 是否使用同一个 embedding 模型。
- 优化 chunk：标题、标签、正文一起进入 embedding 输入。
- 调整向量相似度阈值和 topK。
- 如果向量层不可用，确保系统降级到 ES / FULLTEXT。

### 面试回答

可以这样答：

> 我不会直接看 LLM 最终回答判断 RAG 好坏，而是先把召回链路拆开看。先看 ES 是否召回正确 chunk，再看向量召回，再看 RRF 融合后的 TopK。之前我发现第一版 ES ngram 虽然提高了召回，但也会因为中文短片段命中导致误召回，所以我收紧了 content 字段的匹配策略，提高 ngram 的 minimum_should_match，并提高 title、keywords、tags 的权重。最终像 `Gateway Nacos`、`Redis Lua` 能稳定命中对应文档，而知识库里没有的主题不会再乱召回。

## 2. 如果发现 LLM 响应很慢怎么办？

### 现象

- 用户点发送后很久没有首 token。
- 流式输出中间卡顿。
- 并发 2 到 5 时响应时间明显上升。
- 出现 502、504、SSE 中断。

### 怎么发现

不要只看总耗时，要拆指标：

- 首 token 时间。
- 完整响应时间。
- RAG 检索耗时。
- 上下文构建耗时。
- LLM API 耗时。
- token 数量。
- 并发连接数。
- 错误率。

### 怎么定位

按链路拆：

```text
浏览器
  -> Nginx
  -> Gateway
  -> chatbot-service
  -> 上下文构建
  -> RAG 检索
  -> LLM 调用
  -> SSE 返回
  -> Kafka/Redis 异步记录
```

排查顺序：

1. `/api/chat/health` 是否快，确认基础链路。
2. RAG 检索是否慢，单独测 `/api/knowledge/search` 或 ES 查询。
3. 模型 API 是否慢，看首 token 时间。
4. prompt 是否过长，检查上下文和 RAG TopK。
5. 是否 Ollama 本地模型 CPU/内存不足。
6. 是否 Nginx/Gateway 超时太短。
7. 是否 Kafka/Redis 写入阻塞主链路。

### 怎么解决

- 缩短 prompt：限制上下文长度、RAG TopK、摘要长度。
- 分层上下文：最近历史 + 摘要 + 相关历史，避免全量拼接。
- RAG 检索限时，失败降级。
- 大模型调用设置合理超时。
- 流式返回，尽早输出首 token。
- 本地 Ollama 并发受限时，降低并发或换云端模型。
- 生产环境给 Nginx/Gateway 配置足够长的 stream timeout。
- 异步记录聊天日志，不阻塞主响应。

### 面试回答

> 我会先区分是服务慢还是模型慢。用 `/api/chat/health` 测基础链路，用 RAG debug 测检索耗时，再看 `/api/chat/stream` 的首 token 时间。如果 health 很快但首 token 很慢，通常瓶颈在模型或 prompt 太长。如果 RAG 慢，就优化 ES/PGVector 查询和 TopK。如果是流式中断，就看 Nginx/Gateway 超时。AI 聊天接口的 QPS 不能只看 Spring Boot，通常瓶颈在 LLM 推理和 token 生成。

## 3. 如果上下文丢失或答非所问怎么办？

### 现象

- 用户前面说过的约束，后面回答忘了。
- 长会话后模型只记得最近几轮。
- 上下文拼接太长导致响应慢。

### 怎么发现

查看上下文构建结果：

- 最近历史是否进入 prompt。
- 会话摘要是否存在。
- 相关历史是否选中。
- RAG 引用是否插入。
- prompt 总长度是否超过限制。

相关代码：

- `ChatContextService`
- `ChatbotService`
- `AgentService`

### 怎么解决

- 不只用固定滑动窗口。
- 使用分层上下文：
  - 系统提示词
  - RAG 引用
  - 会话摘要
  - 当前问题相关旧历史
  - 最近原文历史
  - 当前输入
- Redis 保存最近候选历史。
- Redis 保存会话摘要。
- 摘要失败时跳过，不影响主对话。
- 限制 `max-context-chars`，防止 prompt 过长。

### 面试回答

> 旧实现只保留 Redis 最近 5 条历史，长会话容易丢目标和约束。我把它改成分层上下文，由 `ChatContextService` 统一构建，包含最近原文历史、会话摘要、相关历史和 RAG 引用。摘要失败只降级跳过，不影响主链路。这样既保留连续对话，又避免无限拼历史导致 prompt 过长。

## 4. 如果 RAG 召回了内容，但回答仍然胡说怎么办？

### 现象

- 召回结果是对的，但 LLM 回答没用。
- LLM 编造知识库没有的内容。
- 引用和回答不一致。

### 怎么发现

分开看：

1. 检索结果是否正确。
2. prompt 是否真的包含检索结果。
3. 系统提示词是否要求基于资料回答。
4. LLM 是否忽略资料。
5. 资料片段是否太长、太乱、冲突。

### 怎么解决

- prompt 明确要求“优先依据知识库引用”。
- 给 RAG 引用编号。
- 控制每条 snippet 长度。
- 对相似 chunk 去重。
- 提高高质量字段权重，如 title、summary。
- 对无召回结果明确告诉模型“不确定”，不要强答。
- 后续可引入 rerank 模型。

### 面试回答

> RAG 不是召回了就一定回答好。我会先确认召回结果是否正确，再确认 prompt 是否包含这些引用。如果召回对但回答偏，我会优化 prompt 约束、引用格式和 snippet 长度，必要时做 rerank。我的原则是宁可告诉用户知识库没有相关内容，也不要把低相关内容塞给模型导致幻觉。

## 5. 如果 ES 召回很多垃圾结果怎么办？

### 现象

- 查询不存在的主题也返回结果。
- highlight 只有单个字命中。
- 文档标题明显不相关。

### 怎么发现

看 ES 返回：

- score
- title
- highlight
- query tokens
- 命中字段

比如之前测试：

```text
自动续期
```

第一版 ngram 可能因为命中单字或短片段召回无关文档。

### 怎么解决

- standard multi_match 不直接扫全文 content。
- content 主要通过 phrase / ngram 子字段参与。
- ngram 加 `minimum_should_match: 75%`。
- 提高 title / keywords / tags 权重。
- 不相关 query 可以返回空。
- 后续接入 IK analyzer 或自定义词典。

### 面试回答

> 我上线 ES 后做过真实 query 测试。第一版 ngram 提高了召回，但也发现了误召回，比如知识库没有“自动续期”时，也可能因为命中单字召回无关文档。后来我收紧了 content 查询，提高 ngram 的 minimum_should_match，并提高标题和标签权重。调完后，不存在的主题不再乱返回，已有主题能命中正确 chunk。

## 6. 如果 PGVector 不可用怎么办？

### 现象

- 日志出现 `UnknownHostException: pgvector`。
- 向量初始化失败。
- 应用启动被 PGVector 拖垮。

### 怎么发现

看启动日志：

```text
PGVector datasource initialization failed
```

或者 RAG 返回里 vectorResults 为空。

### 怎么解决

- PGVector 初始化失败不能影响主服务启动。
- `PgVectorClient` 标记 unavailable。
- 向量召回返回空列表。
- ES / MySQL FULLTEXT 继续工作。
- 部署环境如果没有 pgvector，就设置 `APP_RAG_VECTOR_ENABLED=false`。

### 面试回答

> PGVector 是增强能力，不应该成为主服务单点故障。我把它改成初始化失败时降级，向量召回返回空，ES 关键词召回和 MySQL FULLTEXT fallback 仍然可用。这样即使向量层挂了，聊天和关键词 RAG 仍然能运行。

## 7. 如果聊天记录偶尔丢失怎么办？

### 现象

- 用户聊完后历史记录缺一条。
- Redis 有，MySQL 没有。
- Kafka 发送失败没有补偿。

### 怎么发现

对比：

- Redis 历史 key。
- MySQL `chat_record`。
- Kafka producer 日志。
- DLT 死信主题。
- outbox 表状态。

相关代码：

- `ChatEventProducer`
- `ChatEventConsumer`
- `KafkaReliabilityTest`
- `chat_event_outbox`

### 怎么解决

- 消费者手动 ACK。
- 成功落库后再 ACK。
- 失败重试。
- 重试失败进入 DLT。
- 发送侧使用 outbox pattern，避免发送失败静默丢失。
- 消费端按 eventId 幂等。

### 面试回答

> 聊天记录不仅是展示历史，还影响上下文和审计，所以不能静默丢。我的做法是 Kafka 消费端手动 ACK，落库成功后再确认；失败交给 DefaultErrorHandler 重试，最终进入 DLT。发送侧进一步用 outbox pattern，把待发送事件先落库，后台补偿发送，避免 producer 发送失败导致消息丢失。

## 8. 如果 Redis 上下文缓存和 MySQL 不一致怎么办？

### 现象

- 页面历史和数据库历史不一致。
- 删除会话后又出现旧上下文。
- Redis 有旧记录。

### 怎么发现

检查：

- `chat:history:{sessionId}`
- `chat:summary:{sessionId}`
- MySQL `chat_record`
- 删除会话接口是否同步删 Redis。

### 怎么解决

- 读：Redis 优先，未命中回源 MySQL。
- 写：Kafka 消费成功后写 MySQL，再更新 Redis。
- 删除：同步删除 history 和 summary key。
- Redis 只做缓存，不做最终事实来源。
- 缓存异常不影响主流程。

### 面试回答

> 我把 Redis 定位为缓存，不作为最终事实来源。聊天历史读取走 cache-aside，未命中回源 MySQL；写入由 Kafka 消费成功后更新 MySQL 和 Redis；删除会话时同步删除 Redis history 和 summary。即使 Redis 失败，也不影响主链路，只是降级回源数据库。

## 9. 如果按钮点击没反应怎么办？

### 现象

- 页面能打开，但按钮没反应。
- Console 报 CSP 错误。
- 例如：
  - `Executing inline script violates Content Security Policy`
  - `Executing inline event handler violates Content Security Policy`

### 怎么发现

打开浏览器 DevTools：

- Console 看 CSP 错误。
- Network 看 JS/CSS 是否被拦截。
- 查看响应头 `Content-Security-Policy`。

### 怎么解决

短期：

- CSP 区分 `script-src-elem` 和 `script-src-attr`。
- 给内联 script 加 nonce。
- 允许现有事件属性过渡运行。
- 去掉不稳定第三方字体资源。

长期：

- 去掉 HTML 内联 `onclick`。
- JS 移到静态文件。
- 使用 addEventListener 绑定。
- CSP 不再允许 inline event。

### 面试回答

> 这个问题不是后端接口挂了，而是浏览器安全策略拦截了前端脚本。页面里大量使用 inline script 和 onclick，但 CSP 没有放行，所以按钮函数根本没执行。我通过 DevTools Console 定位到 CSP 报错，短期用 nonce 修复内联脚本，长期方向是把 JS 外置并移除 onclick。

## 10. 如果 Gateway 返回 502/503/504 怎么办？

### 现象

- 页面能打开，接口 502/503。
- `/api/chat/health` 不通。
- Gateway 日志提示找不到服务实例。

### 怎么发现

按链路检查：

```bash
curl http://127.0.0.1:9000/api/chat/health
docker compose -f docker-compose.prod.yml ps
docker logs --tail 100 chatbot-gateway
docker logs --tail 100 chatbot-service
```

看 Nacos 是否有服务实例。

### 怎么解决

- 确认 chatbot-service 是否启动。
- 确认服务是否注册到 Nacos。
- 确认 Gateway 路由是 `lb://chatbot-service`。
- 确认容器网络可达。
- 流式接口慢时调大 Nginx/Gateway timeout。

### 面试回答

> 我会先用 health 接口判断 Gateway 到服务是否通。如果 health 不通，就看容器状态、Nacos 注册和 Gateway 路由。如果 health 通但业务接口失败，再看业务日志。这样可以先把网络/路由问题和业务问题分开。

## 11. 如果压测 QPS 很低怎么办？

### 现象

- health QPS 正常，但聊天 QPS 很低。
- 并发稍微升高就超时。
- 错误码出现 500/502/504。

### 怎么发现

分层压测：

1. `/api/chat/health`：基础 HTTP 链路。
2. `/api/knowledge/search`：RAG 检索。
3. `/api/chat/stream`：真实聊天。

不要直接用聊天接口判断 Spring Boot QPS，因为它会被模型速度限制。

### 怎么解决

- health 低：看 Gateway/Nginx/Tomcat。
- RAG 慢：看 ES/MySQL/PGVector。
- 聊天慢：看 LLM、prompt 长度、上下文、流式超时。
- 压测时用不同 sessionId，避免上下文越测越长。
- 聊天压测低并发开始，避免烧 token。

### 面试回答

> 我会把 QPS 分层看。health 接口测基础 HTTP 链路，RAG 接口测检索能力，chat stream 才测真实业务。之前 health 在服务器本机 20 并发、200 请求下大约 400 到 450 QPS，但这不代表聊天 QPS。聊天 QPS 主要受 LLM 首 token 时间和生成速度限制，所以要单独看首 token、总耗时和错误率。

## 12. 如果 MySQL 查询变慢怎么办？

### 现象

- Java 关键词检索慢。
- 知识库文档变多后响应变慢。
- 应用 CPU 增加。

### 怎么发现

看旧链路是否把大量文档查出来再 Java contains。

执行计划：

```sql
EXPLAIN SELECT *
FROM knowledge_document
WHERE user_id = ?
  AND enabled = 1
  AND MATCH(title, content, tags) AGAINST (? IN NATURAL LANGUAGE MODE);
```

FULLTEXT 正常时：

```text
type = fulltext
key = ft_knowledge_title_content_tags
```

### 怎么解决

- 给 `title/content/tags` 加 FULLTEXT。
- 用 `MATCH AGAINST` 先做候选粗筛。
- Java 只对最多 200 条候选精排。
- 后续升级 ES 做 chunk 级召回。

### 面试回答

> 我做过一个 1 万条模拟数据的 benchmark。LIKE 扫描和 FULLTEXT 都命中 100 条，但 LIKE 大约 52.9ms，FULLTEXT 大约 1.68ms，快了约 31 倍。FULLTEXT 的作用是把关键词过滤下推到数据库，减少 Java 精排的候选规模。后续引入 ES 是因为 FULLTEXT 仍然是文档级，不能很好支持 chunk 级召回和字段权重。

## 13. 如果文件下载或图片显示 403 怎么办？

### 现象

- 文件上传成功，但图片打不开。
- 浏览器直开下载链接 401/403。
- 聊天图片无法显示。

### 怎么发现

检查：

- Gateway 是否放行下载路径。
- file-service 是否校验 uploaderId。
- 前端是否走 `/api/files/download/**`。
- 是否还硬编码 `:8081`。

### 怎么解决

- 前端统一走 Gateway 相对路径。
- 浏览器直开下载链接时不能依赖 Authorization header。
- 下载路径可在 Gateway 放行，但 file-service 内部必须做权限校验。
- 保证文件记录有 uploaderId。

### 面试回答

> 文件下载和普通 API 不一样，浏览器直开图片时不方便带 Authorization header。我让下载路径走 Gateway，并在 file-service 内部根据 uploaderId 做权限校验，不能只靠 Gateway 放行。这样既能让图片正常显示，又能保持用户隔离。

## 14. 如果 ES 宕机怎么办？

### 现象

- ES health 失败。
- RAG 关键词召回为空。
- 日志出现 ES search failed。

### 怎么发现

```bash
curl http://127.0.0.1:9200/_cluster/health
docker compose -f docker-compose.prod.yml ps elasticsearch
docker logs --tail 100 chatbot-elasticsearch
```

### 怎么解决

- ES 查询失败返回空，不抛到主聊天链路。
- 回退 MySQL FULLTEXT。
- FULLTEXT 再失败，回退最近文档有限扫描。
- 4GB 服务器内存紧张时可关闭 ES：
  `APP_RAG_ELASTICSEARCH_ENABLED=false`。

### 面试回答

> ES 是关键词召回增强层，不应该影响聊天主链路。我保留了 fallback：ES 失败回退 MySQL FULLTEXT，FULLTEXT 失败再做有限扫描。这样即使 ES 在 4GB 单机上因为内存压力不可用，系统仍然能回答，只是召回质量下降。

## 15. 如果测试环境和线上结果不一致怎么办？

### 现象

- 本地能登录，线上按钮没反应。
- 本地 ES 有结果，线上无结果。
- 本地接口通，线上 502。

### 怎么发现

对比四类信息：

- 环境变量：`docker exec chatbot-service printenv`
- compose：`docker compose -f docker-compose.prod.yml config`
- 实际容器状态：`docker compose ps`
- 浏览器 Console / Network

### 怎么解决

- 不只看代码，要看线上运行配置。
- 确认服务是否重建。
- 确认浏览器缓存。
- 确认 Nginx/Gateway 是否指向新服务。
- 确认 ES index 是否重建。
- 确认 CSP、域名、HTTPS、Cloudflare 注入脚本。

### 面试回答

> 我会区分代码状态和线上运行状态。比如本地已经改了 CSP，但如果远程镜像没重建，线上仍然是旧响应头。排查时我会看容器启动时间、环境变量、实际响应头和浏览器 Console，而不是只看代码。这样能避免“代码看起来对，但线上没生效”的误判。
